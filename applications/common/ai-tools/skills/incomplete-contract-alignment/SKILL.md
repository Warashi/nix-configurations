# Incomplete Contract Alignment (不完備契約アプローチ)

このスキルは、仕様に明記されていない事態やトレードオフが発生した際、エージェントが自律的かつ一貫した判断を下せるようにするための「憲法（設計思想）」をユーザーと合意し、明文化する手順を定義します。

## 実行トリガー

以下のいずれかの状況でこのスキルを実行してください：
* 新規プロジェクトや新機能の実装を開始する直前
* 既存コードの設計方針を大きく変更（リファクタリング）する時
* ユーザーの要件が曖昧で、複数の実装方針（パフォーマンス重視、堅牢性重視など）が考えられる時

## ワークフロー (合意形成手順)

以下のステップを順に実行してください：

- [ ] **Step 1: 意図の分析**
      ユーザーの初期要件から、暗黙的に求められている「コア・フィロソフィー（最優先事項）」「制約」「拡張性の方向」を推測する。
- [ ] **Step 2: 契約の提案**
      下記の「出力テンプレート」を使用し、ユーザーに設計方針を提案して合意を求める。
- [ ] **Step 3: 修正と合意**
      ユーザーからフィードバックがあれば内容を修正し、ユーザーが明確に承認するまでイテレーションを行う。
- [ ] **Step 4: 憲法の永続化 (重要)**
      合意が形成されたら、その内容を `docs/PROJECT_CONSTITUTION.md` （またはそれに類するドキュメントディレクトリ）としてファイルに書き出して保存する。

## 出力テンプレート

Step 2でユーザーに提案する際は、以下のフォーマットを使用してください：

```text
実装を開始する前に、本プロジェクトにおける「設計思想（憲法）」をすり合わせさせてください。
以下の定義で合意（契約）してよろしいでしょうか？

1. 【コア・フィロソフィー（最優先事項）】: [例：実行速度よりも開発者の理解しやすさと保守性を優先する]
2. 【不変の制約】: [例：外部ライブラリへの依存は最小限に留め、標準APIをベースにする]
3. 【拡張性の解釈】: [例：将来的にマイクロサービス化しやすいよう、ビジネスロジックは疎結合に保つ]

この方針に同意いただけるか、あるいは修正・追加が必要な箇所があれば教えてください。
合意いただけた場合、これを `PROJECT_CONSTITUTION.md` として保存し、以後の自律的な判断基準とします。
```

## Gotchas (注意点・エッジケース)

* **軽微なタスクでは発動しない:** 単一のバグ修正や、数行のコード追加などのコンテキストが明らかな小さなタスクでは、この合意プロセスはスキップしてください。
* **事後検証 (Validation):** 合意（Step 4）以降にコードを生成する際は、必ず事前に `docs/PROJECT_CONSTITUTION.md` を読み込み、自分の実装計画が憲法に違反していないかセルフチェックを行ってください。
* **ユーザーの契約違反への対処:** 実装フェーズにおいて、ユーザーから「憲法に違反する要求（例：保守性優先で合意したのに、可読性を破壊するハックを要求されるなど）」があった場合は、直ちに実装せず、「この変更は合意した憲法（[違反している項目]）と対立しますが、憲法をアップデート（破棄）して実装を進めますか？」と確認を取ってください。
